### 📋 任务规划框架（5 步法）

1. **需求分析** - 输入/输出/约束/验收
2. **任务分解** - 任务链（ID + 工作内容 + 依赖 + 产出）
   - **每个主任务细化为 1-2 小时的分支子任务**
   - 格式：`T2.1`、`T2.2`、`T2.3`...
   - 明确前置依赖、验收标准
   - **细化到具体执行步骤（像搭乐高）**
3. **执行计划** - 详细步骤
4. **验证方案** - 方法 + 标准
5. **风险预案** - 风险 + 概率 + 影响 + 应对

### 🎯 任务细化原则（搭乐高式）

**为什么要细化到具体步骤？**
- ✅ **准确定位错误** - 出问题知道具体哪一步
- ✅ **降低难度** - 大任务拆成小积木，逐个击破
- ✅ **易于验证** - 每步都有明确验收标准
- ✅ **减少返工** - 每步验证通过再下一步

**细化标准：**
- 具体要创建/修改的文件路径
- 具体要实现的函数/类名
- 具体的输入输出
- 具体的验收标准

**每个子任务必须细化到：**
- 具体要创建/修改的文件路径
- 具体要实现的函数/类名
- 具体的输入输出
- 具体的验收标准

### 🎯 乐高式开发模式（核心）

```
项目框架 → 任务链 → 子任务 → 步骤 → 验证 → 下一任务
```

#### 1. 程序搭框架
- 程序负责整体项目架构设计
- 将项目拆分为多个"任务块"（如 T1、T2、T3...）

#### 2. 任务链设计
- 每个任务块包含若干"子任务"（如 T1.1、T1.2、T1.3...）
- 每个子任务**必须细化到具体执行步骤**

#### 3. 子任务验证流程（每个子任务必须执行）
```
子任务开始 → 执行 → 自我验证 → 提交报告 → 技术验证 → 视觉验证 → 通过 → 下一子任务
                                        ↓未通过
                                     打回改进
```

#### 4. 任务块验证
- 当一个任务块的所有子任务完成后
- 进行**任务块级别的双验证**
- 通过后才能进入下一个任务块
