# 问题解决机制

> 明确问题升级路径和最终解决者

---

## 问题解决层级

```
发现问题
    ↓
第一层: 开发员尝试解决
    ↓ 无法解决
第二层: 主程序介入解决
    ↓ 无法解决
第三层: 架构师提供方案
    ↓ 无法解决
第四层: 总管决策
```

---

## 问题分类与解决者

### 1. 代码问题

| 问题类型 | 第一解决者 | 最终解决者 |
|----------|------------|------------|
| 编译错误 | 开发员 | **主程序** |
| 测试失败 | 开发员 | **主程序** |
| 代码质量问题 | 开发员 | **主程序** |
| 架构不符合 | 主程序 | **主程序** |
| 性能问题 | 主程序 | **主程序** |

**流程**:
```
开发员发现问题 → 尝试解决
    ↓ 无法解决
上报主程序 → 主程序解决
    ↓ 架构问题
上报架构师 → 架构师提供方案 → 主程序实施
```

### 2. 功能操作问题

| 问题类型 | 第一发现者 | 验证者 | 最终解决者 |
|----------|------------|--------|------------|
| UI 显示异常 | 美术 | 美术/主管 | **主程序** |
| 交互逻辑错误 | 美术 | 美术/主管 | **主程序** |
| 功能不符合需求 | 主管 | 主管 | **主程序** |
| 视觉风格偏差 | 美术 | 美术 | **主程序** |

**流程**:
```
美术/主管发现问题 → 记录问题
    ↓
通知主程序 → 主程序分析
    ↓
主程序修复 → 美术/主管验证
    ↓
验证通过/不通过 → 继续修复
```

### 3. 架构设计问题

| 问题类型 | 发现者 | 解决方案提供者 | 实施者 |
|----------|--------|----------------|--------|
| 模块划分不合理 | 主程序 | 架构师 | **主程序** |
| 接口设计缺陷 | 主程序 | 架构师 | **主程序** |
| 技术选型错误 | 主程序 | 架构师 | **主程序** |

**流程**:
```
主程序发现架构问题
    ↓
咨询架构师 → 架构师提供新方案
    ↓
主程序实施修改
```

### 4. 流程执行问题

| 问题类型 | 发现者 | 解决者 |
|----------|--------|--------|
| 流程违规 | 主管 | 主管纠正 |
| 验证不通过 | 任何角色 | **主程序**修复 |
| 进度延迟 | 主管 | 主管调整/主程序加速 |

## 最终解决者规则

### 主程序是最终技术解决者

**所有技术问题最终都由主程序解决**:
- ✅ 代码问题 → 主程序修复
- ✅ 功能问题 → 主程序修复
- ✅ 架构实施 → 主程序执行
- ✅ 性能优化 → 主程序负责

**开发员职责**:
- 尝试解决自己负责模块的问题
- 无法解决时立即上报主程序
- 不拖延，不隐藏问题

**主程序职责**:
- 及时响应开发员上报的问题
- 提供解决方案或修复代码
- 确保问题彻底解决
- 记录问题和解决方案

### 问题解决时限

| 问题级别 | 解决时限 | 升级条件 |
|----------|----------|----------|
| 阻塞开发 | 30分钟 | 超时升级架构师 |
| 影响功能 | 2小时 | 超时升级架构师 |
| 代码质量 | 当天 | 超时升级主管 |
| 轻微优化 | 本周 | 不紧急 |

---

## 关键原则

1. **不拖延**: 发现问题立即上报
2. **不隐藏**: 不掩盖问题，及时暴露
3. **主程序兜底**: 所有技术问题最终由主程序解决
4. **验证闭环**: 解决问题后必须验证
5. **记录归档**: 重要问题记录到 MEMORY.md
