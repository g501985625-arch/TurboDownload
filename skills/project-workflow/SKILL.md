## Phase 0: 需求分析（自动化）

```yaml
触发条件: 新项目启动信号 / 用户输入需求
执行者: 自动化主管

步骤:
  1. 收集需求:
     - 自动解析用户输入
     - 提取功能需求
     - 提取非功能需求
     
  2. 生成需求文档:
     - 创建 docs/REQUIREMENTS.md
     - 功能列表
     - 约束条件
     - 验收标准
     
  3. 自动触发 Phase 1

输出: REQUIREMENTS.md
```

## Phase 1: 架构设计（自动化触发 + 架构师执行）

### 1.1 自动触发架构设计

```yaml
触发条件: REQUIREMENTS.md 生成完成
执行者: 自动化主管

步骤:
  1. 自动发送任务给架构师:
     - 需求文档路径
     - 设计模板
     - 验收标准
     
  2. 监控设计进度:
     - 每30分钟检查一次
     - 自动提醒超时
     
  3. 设计完成后:
     - 自动验证设计文档完整性
     - 自动触发 Phase 1.2
```

### 1.2 框架搭建（架构师）

```yaml
执行者: 架构师
任务: 设计整体框架
输出:
  - design/ARCHITECTURE.md
  - 技术栈选型
  - 模块依赖关系
  - 接口定义

自动化:
  - 自动检查文档完整性
  - 自动验证设计合理性
  - 自动触发 Phase 1.3
```

### 1.3 模块划分（架构师）

```yaml
执行者: 架构师
任务: 划分功能模块
输出:
  - 模块列表
  - 模块职责
  - 模块接口
  - 依赖关系图

自动化:
  - 自动检查模块划分完整性
  - 自动验证模块耦合度
  - 自动触发 Phase 1.4
```

### 1.4 详细任务链设计（架构师）

```yaml
执行者: 架构师
任务: 设计详细任务链
输出:
  - planning/P{模块}/DETAILED_TASK_CHAIN.md
  - 每个子任务 1-2 小时
  - 明确依赖关系
  - 明确验收标准

自动化:
  - 自动解析任务链
  - 自动识别可并行任务
  - 自动生成任务分配表
  - 自动触发 Phase 2
```

## 问题解决机制（重要）

**主程序是最终技术解决者**

所有问题按以下路径解决：

| 问题类型 | 第一解决者 | 最终解决者 | 文档 |
|----------|------------|------------|------|
| 代码问题 | 开发员 | **主程序** | PROBLEM_RESOLUTION.md |
| 功能操作问题 | 美术/主管 | **主程序** | PROBLEM_RESOLUTION.md |
| 架构设计问题 | 主程序 | 架构师提供方案，主程序实施 | PROBLEM_RESOLUTION.md |

**核心原则**: 所有技术问题最终都由主程序解决。

详细机制见: `skills/project-workflow/PROBLEM_RESOLUTION.md`

## Phase 2: 并行开发（自动化）

### 2.1 自动任务分配

```yaml
触发条件: 任务链设计完成
执行者: 自动化主管

步骤:
  1. 解析任务链:
     - 识别核心模块 → 主程序
     - 识别基础模块 → 开发员
     - 识别可并行任务
     
  2. 自动分配:
     - 发送任务给主程序
     - 发送任务给开发员
     
  3. 记录分配状态
```

### 2.2 自动并行开发监控

```yaml
执行者: 自动化主管
监控频率: 每5分钟

监控内容:
  - 代码提交状态
  - 编译状态 (cargo check)
  - 测试状态 (cargo test)
  - 子代理响应状态

自动处理:
  - 编译失败 → 自动通知修复
  - 测试失败 → 自动通知修复
  - 代理超时 → 自动提醒/重分配
```

### 2.3 子任务级互相验证（新增）

```yaml
触发条件: 子任务完成（双方都有产出）
执行者: 自动化主管

步骤:
  1. 自动触发互相审查:
     - 主程序审查开发员的子任务代码
     - 开发员审查主程序的子任务代码
     
  2. 收集审查意见:
     - 代码质量评估
     - 问题记录
     - 改进建议
     
  3. 判断结果:
     - 通过 → 继续下一子任务
     - 不通过 → 自动分配修复任务

注意:
  - 子任务级验证是轻量级快速验证
  - 模块级验证（Phase 3）是深度全面验证
```

### 2.4 子任务级视觉验证（新增，按需）

```yaml
触发条件: 子任务涉及UI/视觉元素
执行者: 自动化主管 + 美术

自动化部分:
  - 自动截图UI界面
  - 自动对比设计规范
  - 自动检查视觉问题

美术部分:
  - 快速审查视觉风格
  - 确认是否符合设计
  - 记录视觉问题

判断:
  - 有UI元素 → 触发视觉验证
  - 无UI元素（纯后端） → 跳过

注意:
  - 子任务级视觉验证是快速检查
  - 模块级视觉验证（Phase 3）是全面审查
```

### 2.5 自动汇报收集

```yaml
触发条件: 子代理发送汇报（包含互相验证结果）
执行者: 自动化主管

步骤:
  1. 接收汇报
  2. 解析内容:
     - 任务ID
     - 完成状态
     - 自我验证结果
     - 互相验证结果
     - 视觉验证结果（如有）
  3. 更新进度看板
  4. 如模块完成:
     - 自动触发 Phase 3（模块级深度验证）
```

## Phase 3: 模块验证（自动化）

### 3.1 自动互相验证

```yaml
触发条件: 模块所有子任务完成
执行者: 自动化主管

步骤:
  1. 自动触发:
     - 发送审查指令给主程序
     - 发送审查指令给开发员
     
  2. 收集验证报告:
     - 等待双方提交
     - 解析验证结果
     
  3. 判断结果:
     - 通过 → 触发 Phase 3.2
     - 不通过 → 自动分配修复任务
```

### 3.2 整体技术验证（自动化 + 架构师）

```yaml
触发条件: 互相验证通过
执行者: 自动化主管 + 架构师

自动化部分:
  - 自动运行完整测试套件
  - 自动检查代码覆盖率
  - 自动检查架构符合性
  - 自动生成技术验证报告

架构师部分:
  - 审查整体架构实现
  - 确认接口一致性
  - 确认设计目标达成
  
输出: 技术验证报告
```

### 3.3 视觉操作验证（自动化 + 美术）

```yaml
触发条件: 技术验证通过
执行者: 自动化主管 + 美术

自动化部分:
  - 自动截图 UI 界面
  - 自动检查视觉规范
  - 自动对比设计稿

美术部分:
  - 审查视觉风格一致性
  - 审查色彩搭配
  - 审查像素风强度
  - 确认操作流畅性
  
输出: 视觉验证报告
```

## Phase 5: 交付验收（自动化）

### 5.1 自动生成交付报告

```yaml
触发条件: 所有测试通过
执行者: 自动化主管

报告内容:
  - 项目概述
  - 功能清单
  - 测试报告汇总
  - 性能指标
  - 已知问题
  - 使用说明
  - 安装包

输出: docs/DELIVERY_REPORT.md
```

### 5.2 自动交付

```yaml
执行者: 自动化主管

步骤:
  1. 自动打包:
     - 源代码
     - 二进制文件
     - 文档
     - 安装包
     
  2. 自动发布:
     - 上传到交付位置
     - 生成下载链接
     - 发送通知给用户
```

### 5.3 用户验收

```yaml
执行者: 用户
自动化支持:
  - 提供验收清单
  - 收集用户反馈
  - 记录问题
  - 自动分配修复任务（如有问题）

结果:
  - 验收通过 → 项目完成
  - 验收不通过 → 自动触发修复流程
```
