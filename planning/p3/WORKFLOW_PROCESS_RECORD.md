## 一、角色与职责定义

### 1.1 团队架构

```
你（人类总管）→ 顶层需求
    ↓
【千问 3.5 Plus】→ 总管（监督 + 最终验证）
    ├─ 技术验证：审查代码逻辑、架构合理性
    ├─ 视觉验证：审查截图、UI 效果、风格一致性
    ├─ 流程监督：确保 v2.1 流程严格执行
    └─ 子代理管理：启动、监督、验收子代理任务
        ↓
    【GLM-5/MiniMax-M2.5】→ 架构师（技术架构 + 任务规划）
        ├─ 系统设计：模块划分、接口定义、数据流
        ├─ 技术约束：制定开发规范
        ├─ 任务链规划：搭乐高式细化到1-2小时子任务
        └─ 代码审查：技术验证
            ↓
        【GLM-5】→ 主程序（任务分配 + 核心开发）
            ├─ 接收任务链，拆分子任务
            ├─ 分配子任务给开发员
            ├─ 开发核心模块
            └─ 代码审查开发员产出
                ↓
            【GLM-5/MiniMax-M2.5】→ 开发员（具体执行）
                ├─ 接收主程序分配的具体子任务
                ├─ 按任务要求开发
                └─ 边写边测，自我验证
```

## 三、v2.1 流程执行步骤

### 3.1 标准执行流程

```
Phase 0: 需求分析
    ↓
Phase 1: 架构设计（架构师）
    - 框架搭建 → 模块划分 → 详细任务链设计
    ↓
Phase 2: 并行开发（主程序 + 开发员）
    - 总管启动子代理
    - 子代理并行执行
    - 自动分配 → 并行开发 → 子任务级互相验证 → 自动汇报
    ↓
Phase 3: 模块验证
    - 模块级互相验证 → 整体技术验证 → 视觉操作验证（按需）
    ↓
Phase 4: 整合测试
    - 自动集成 → 功能测试 → 性能测试 → 端到端测试
    ↓
Phase 5: 交付验收
    - 自动生成报告 → 自动交付 → 用户验收
```

### 3.2 互相验证流程

```
1. 开发员完成代码 → 自我验证 → 提交报告
2. 主程序完成代码 → 自我验证 → 提交报告
3. 开发员审查主程序代码 → 生成审查报告
4. 主程序审查开发员代码 → 生成审查报告
5. 总管审查两份报告 → 生成最终审查报告
6. 批准进入下一阶段
```

### 3.3 文档生成要求

| 文档 | 生成者 | 内容 |
|------|--------|------|
| `VERIFICATION_REPORT_DEVELOPER.md` | 开发员 | 审查主程序代码报告 |
| `VERIFICATION_REPORT_LEAD.md` | 主程序 | 审查开发员代码报告 |
| `VERIFICATION_REPORT_PM.md` | 总管 | 最终审查报告 |
| `WORKFLOW_PROCESS_RECORD.md` | 总管 | 流程执行记录（本文档） |

## 四、关键规则（v2.1 强制）

1. ✅ **每个子任务完成 → 立即汇报**（禁止累积多个任务后汇报）
2. ✅ **子任务级互相验证**（主程序审开发员，开发员审主程序）
3. ✅ **子任务级视觉验证**（按需，纯后端模块跳过）
4. ✅ **模块级深度验证**（所有子任务完成后）
5. ✅ **阶段开始前必须完成详细任务链设计**
6. ✅ **总管必须启动子代理执行任务，不得自己直接开发**
7. ✅ **每次子代理启动必须记录到本文档**
