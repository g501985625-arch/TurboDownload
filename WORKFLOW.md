## 🏗️ 五层分工架构

```
你（人类总管）→ 顶层需求
    ↓
【千问 3.5 Plus】→ 总管（监督 + 最终验证）
    ├─ 技术验证：审查代码逻辑、架构合理性
    ├─ 视觉验证：审查截图、UI 效果、风格一致性
    └─ 进度把控
        ↓
    【GLM-5/MiniMax-M2.5】→ 架构师（技术架构 + 任务规划）
        ├─ 系统设计：模块划分、接口定义、数据流
        ├─ 技术约束：制定开发规范
        ├─ 任务链规划：搭乐高式细化到1-2小时子任务
        └─ 代码审查：技术验证
            ↓
        【GLM-5】→ 主程序（任务分配 + 核心开发）
            ├─ 接收任务链，拆分子任务
            ├─ 分配子任务给开发员
            ├─ 开发核心模块
            └─ 代码审查开发员产出
                ↓
            【GLM-5/MiniMax-M2.5】→ 开发员（具体执行）
                ├─ 接收主程序分配的具体子任务
                ├─ 按任务要求开发
                └─ 边写边测，自我验证
                    ↓
                【Kimi K2.5】→ 美术（仅出图）
```

### 角色职责

| 角色 | 代理 | 模型 | 职责 |
|------|------|------|------|
| **总管** | executive-producer | 千问 3.5 Plus | 监督 + 最终验证（技术 + 视觉） |
| **架构师** | chief-architect | GLM-5 / MiniMax-M2.5 | 技术架构 + 任务规划 + 代码审查 |
| **主程序** | lead-developer | GLM-5 | 任务分配 + 核心开发 + 审查开发员 |
| **开发员** | developer | Qwen3-Coder-Plus | 具体子任务 + 审查主程序代码 |
| **美术** | art-director | Kimi K2.5 | 仅出图（按需视觉验证） |

## 📋 完整自动化流程（Phase 0-5）

```
Phase 0: 需求分析
    ↓
Phase 1: 架构设计（架构师）
    - 框架搭建 → 模块划分 → 详细任务链设计
    ↓
Phase 2: 并行开发（主程序 + 开发员）
    - 自动分配 → 并行开发 → 子任务级互相验证 → 自动汇报
    ↓
Phase 3: 模块验证
    - 模块级互相验证 → 整体技术验证 → 视觉操作验证（按需）
    ↓
Phase 4: 整合测试
    - 自动集成 → 功能测试 → 性能测试 → 端到端测试
    ↓
Phase 5: 交付验收
    - 自动生成报告 → 自动交付 → 用户验收
```

## 🔄 验证流程

### 子任务级验证（轻量级，每个子任务完成后）

```
主程序完成子任务 → 自我验证 → 开发员审查
开发员完成子任务 → 自我验证 → 主程序审查
```

### 模块级验证（深度，所有子任务完成后）

```
模块完成
    ↓
主程序 ↔ 开发员 互相技术验证（深度）
    ↓
架构师整体技术验证
    ↓
美术视觉验证（按需）
    ↓
总管最终确认
```

## ⚡ 关键规则（v2.1 强制）

1. ✅ **阶段开始前必须完成详细任务链设计**
2. ✅ **每个子任务完成 → 立即汇报**（禁止累积多个任务后汇报）
3. ✅ **子任务级互相验证**（主程序审开发员，开发员审主程序）
4. ✅ **子任务级视觉验证**（按需，纯后端模块跳过）
5. ✅ **模块级深度验证**（所有子任务完成后）
6. ✅ **任务细分** - 1-2 小时/子任务
7. ✅ **每步验证** - 自我验证 + 互相验证
8. ✅ **规划优先** - 没有详细任务链禁止开发
